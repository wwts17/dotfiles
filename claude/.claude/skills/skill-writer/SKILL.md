# Skill 编写规范

## 核心
同类型 skill 套同一套模板,前两节(`核心`/`适用场景`)所有 skill 一致,后两节按类型分;格式统一才能一眼看出写得好不好。

## 适用场景
新增、改写或评审任何 skill 时。先选模板,再逐节填,最后照"命名规范"核对。

## 选哪套模板
- **分析框架 / 思维方法**(MECE、第一性原理……)→ 模板 A
- **代码风格、提交、流程约定等偏好**→ 模板 B
- 两者都不属于:就近选一套,在 review 时说明为什么

## 模板 A — 思维工具
| 节 | 内容 |
|---|---|
| `## 核心` | 1 句话,工具的核心思想 |
| `## 适用场景` | 触发信号 + 一行"不用"边界 |
| `## 思考步骤` | 编号步骤,动词开头,每步一句 |
| `## 输出结构` | 一个填好的具体范例,展示用完应产出什么形状的结论 |

## 模板 B — 规范/偏好
| 节 | 内容 |
|---|---|
| `## 核心` | 1 句话,这条规范要保证什么 |
| `## 适用场景` | 触发场景(写/评审代码时、提交时) |
| `## 规则` | 要点或表格,每条尽量配 ❌/✅ 对照 |
| `## 范例` | 一个完整正确示例 |

## frontmatter 规范
- 只要 `name` 和 `description` 两个字段
- `name`:小写字母/数字/连字符;不含 `claude`/`anthropic`;与目录名一致
- `description`:第三人称;写清"做什么 + 何时用",含触发关键词;≤1024 字
- 正文 <500 行(思维工具目标 ≤35);只写 Claude 不知道的;引用文件不超一层

## 命名规范
- 用名词短语:`mece-decomposition`、`git-commits`
- 避免含糊词:`helper`、`utils`、`deep-*`、`tools`
- 仅 ASCII(中文写在 description 和正文里)
