# 产出质量评审

## 核心
先判类型,再套该类型 rubric,逐维度先列具体证据、再据此定档;不给连续数字分。理由:大刻度绝对分(尤其 0-100)在 LLM 上跨次漂移大,且受 rubric 顺序、分值标号等偏差影响;少档 + 每档锚定描述更稳。

## 适用场景
评审一份产出,或交付重要产出前自检时。手动 `/quality-review`,或由 CLAUDE.md 在交付前指向它。
不用:闲聊、临时草稿、一次性脚本——评审成本高于收益。

## 怎么评
1. 判类型:下面四类之一;都不属于就近选一类并说明
2. 取该类型 rubric,逐维度核对
3. 逐维度独立核对:每维度先写下指到行/段的具体问题(证据),再据此定档;不让某一维度的好坏带偏其他维度,不先定档再补理由
4. 汇总粗判:良好 / 待改 / 不及格;判不及格必须能点出阻断项

## 三档含义(按此判,不凭整体印象)
- **良好**:该维度可直接交付;挑不出阻断项,至多是吹毛求疵
- **待改**:能交付,但有明确该修的点,修了更好;非阻断
- **不及格**:存在至少一个阻断项——使产出不可用、误导读者、或无法照着做成

## 各类型 rubric

### SKILL.md / prompt
- 直接对照 `skill-writer` 的模板和命名规范(不重复)
- 另查:`description` 触发是否精准;正文是否只写 Claude 不知道的;有无占位符或生造黑话

### 技术文档(README / API / 设计)
- 结构:是否符合 Diátaxis 类型(教程/操作/参考/解释),单一职责不混类
- 可上手:读者能照着做成;前置条件和示例齐全
- 准确:命令、路径、版本真实可跑,无过期信息
- 简洁:无废话、无与代码重复的内容

### 计划 / 方案
- 有 Context:为什么做、要什么结果
- 具体:点名要改的文件、可执行,不是空泛大纲
- 可验证:写明怎么测/怎么确认做成了
- 取舍清楚:只留推荐方案 + 关键风险

### 代码 / 提交信息
- 代码:对照 `code-style`(清晰胜过聪明、错误显式、不过早抽象、零值可用)
- 提交:对照 `git-commits`(Conventional Commits、正文说为什么、无 AI 署名)

## 输出格式
```
类型:<判定>
- <维度>:良好 | 待改 | 不及格 — <具体问题,指到行/段>
- ...
粗判:良好 | 待改 | 不及格
阻断项(仅不及格):<...>
```
维度处于两档之间、拿不准时,写明"判断点 + 依据",不假装确定。
